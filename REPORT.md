## 1. Реализация Redis-кэша
Реализован redis кэш для двух сущностей:
- каталог товаров
- карточка конкретного заказа

Ключи:
- catalog:v1 — кэш каталога
- order_card:v1:{order_id} — кэш карточки заказа

TTL: 60 сек

Нужно указать:
- ключ каталога;
- ключ карточки заказа;
- TTL;
- логику cache hit/miss.

Логика cache hit/miss:
1. При `use_cache=true` сначала выполняется get из Redis
2. Если key найден, hit — данные сразу возвращаются клиенту
3. Если key не найден, miss — данные читаются из БД, сериализуются в JSON и сохраняются в redis
4. При `use_cache=false` данные всегда берутся напрямую из БД без обращения к redis

## 2. Демонстрация неконсистентности (намеренно сломанный сценарий)
Сценарий реализован endpoint:
- `POST /api/cache-demo/orders/{id}/mutate-without-invalidation`

Шаги:
1. Прогрев кэша: `GET /api/cache-demo/orders/{id}/card?use_cache=true`
2. Изменение суммы заказа в бд через `mutate-without-invalidation`
3. Повторный запрос карточки с `use_cache=true`

Результат:
- в бд сумма уже новая
- в ответе API возвращается старая сумма из Redis,так как ключ карточки не инвалидируется

Минимум:
1. Прогрев кэша.
2. Изменение заказа в БД без инвалидации.
3. Повторный запрос из кэша.
4. Чем stale ответ отличается от данных в БД.

## 3. Починка через событийную инвалидацию
- событие `OrderUpdatedEvent(order_id)` создаётся после обновления заказа

- `CacheInvalidationEventBus.publish_order_updated(...)` обрабатывает событие и вызывает инвалидацию кэша

Точка генерации события:
- `POST /api/cache-demo/orders/{id}/mutate-with-event-invalidation`

Точка обработки события:
- `backend/app/application/cache_events.py`

Какие ключи инвалидируются:
- `order_card:v1:{order_id}` — всегда,так как карточка заказа изменена
- `catalog:v1` — тоже инвалидируется, чтобы избежать рассинхронизации агрегатов каталога

Нужно указать:
- где генерируется событие изменения заказа;
- где обрабатывается событие;
- какие ключи инвалидируются и почему.

## 4. Rate limiting endpoint оплаты через Redis
Реализован middleware `RateLimitMiddleware`:
- `/api/orders/{order_id}/pay`
- `/api/payments/retry-demo`

политика:
- по умолчанию 5 запросов за 10 секунд

Ключ лимита:
- `rate_limit:pay:{subject}`
- `subject = user_id` (если есть заголовок `X-User-Id`), иначе IP клиента

Алгоритм:
1. `INCR key`
2. Если это первый запрос в окне — `EXPIRE key window_seconds`
3. Если счётчик > лимита — `429 Too Many Requests`
4. Иначе запрос проходит.

Заголовки:
- `X-RateLimit-Limit`
- `X-RateLimit-Remaining`
- `X-RateLimit-Reset`

Нужно указать:
- policy (например N запросов за M секунд);
- ключ лимита (по user_id/ip);
- поведение при превышении (`429`);
- какие заголовки возвращаются клиенту.

## 5. Бенчмарки RPS до/после кэша
Схема:
- 4 потока, 100 соединений/users, длительность 30 секунд
- сравнение `use_cache=false` и `use_cache=true` на одинаковом `order_id`.

1) Endpoint: `/api/cache-demo/catalog`

- С кэшем: RPS **1064.14**, avg latency **94.25 ms**, timeout **0**

2) Endpoint: `/api/cache-demo/orders/{id}/card`

## 6. Выводы
1. Кэш Redis заметно снижает нагрузку на БД для часто читаемых endpoint, особенно для агрегаций и повторяющихся карточек.
2. Основная сложность кэша - не чтение/запись, а консистентность, без инвалидации stale data появляется сразу.
3. Rate limiting нужен даже при корректной бизнес-логике оплаты, он защищает от повторных кликов,простых DDoS-сценариев.
